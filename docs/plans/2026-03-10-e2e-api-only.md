## Summary

**Goal:** Remove all direct Prisma/database access from E2E tests by adding admin API endpoints for user role management and transaction seeding, then refactoring the entire test suite to use API-only setup. This fixes 39 skipped tests and enables full external E2E coverage against production.

**Architecture:** Add two new admin endpoints — `PATCH /api/admin/users/:id` (set role flags) and `POST /api/admin/e2e/seed-transaction` (create test transaction, guarded by `E2E_SEED_ENABLED` env var). Add a cleanup endpoint `DELETE /api/admin/e2e/cleanup` to remove test data via API. Refactor E2E helpers to use these endpoints instead of Prisma. Remove all `PrismaClient` imports, `setupComplete` flags, `rolesSet` checks, and `ctx.skip()` patterns from test files. Projects are seeded via `POST /api/projects` + `POST /api/projects/:id/publish` (no admin approve needed — `isApproved: true` is set at creation).

**Tech Stack:** Next.js 15 API routes, Prisma, Vitest, Zod validation

## Scope

### In Scope

- `PATCH /api/admin/users/:id` — set `isSeller`, `isVerifiedSeller`, `isAdmin` flags
- `POST /api/admin/e2e/seed-transaction` — create a completed transaction for reviews testing (guarded by `E2E_SEED_ENABLED`)
- `DELETE /api/admin/e2e/cleanup` — delete all test data with `e2e_test_` prefix via API
- Refactor `tests/e2e/helpers.ts` to replace Prisma calls with admin API calls
- Refactor all 7 E2E test files (suites 03-09) to remove Prisma and skip patterns
- Unit tests for all new endpoints

### Out of Scope

- Suite 01 (Health) and Suite 02 (User Identity) — already work without Prisma
- Admin dashboard UI for the new endpoints

- **Domain context:**
  - Users have role flags: `isSeller`, `isVerifiedSeller`, `isAdmin`, `isBuyer`

  - Projects start as `draft` with `isApproved: true`. Publishing sets `status: 'active'`. No admin approval step needed for normal flow.
  - Transactions require: `projectId`, `sellerId`, `buyerId`, `amountCents`, `commissionCents`, `sellerReceivesCents`, `paymentStatus`, `escrowStatus`, `stripePaymentIntentId`, `stripeChargeId`, `codeDeliveryStatus`, `escrowReleaseDate`

## Assumptions

- `requireAdminApiAuth` works with API keys (Bearer token) for admin users — supported by `lib/api-auth.ts:66-81` (Authorization header path) and `lib/api-auth.ts:100` (checks `auth.user.isAdmin`). Tasks 1, 2 depend on this.
- `ProjectService.createProject()` sets `isApproved: true` at line 201 — published projects are automatically approved and visible in public listings without an admin approve call. Tasks 3, 4, 5, 6 depend on this.
- `AdminService.approveProject()` rejects `status === 'active'` at line 277 — cannot approve already-published projects. Must use draft projects for suite 09 approve tests. Task 6 depends on this.
- An admin user with a pre-provisioned API key exists in production (env var `E2E_ADMIN_API_KEY`). Tasks 3-6 depend on this.
- Firebase Admin SDK operations (user creation, token exchange, session creation) work without database access — supported by `tests/e2e/helpers.ts:142-166` (Firebase + API calls only). Task 3 depends on this.

## Risks and Mitigations

| Risk                                                                   | Likelihood | Impact                                    | Mitigation                                                                                                                         |
| ---------------------------------------------------------------------- | ---------- | ----------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- |
| `E2E_ADMIN_API_KEY` not set in Railway                                 | Medium     | All E2E tests that need admin API fail    | Startup guard in helpers.ts throws clear error with setup instructions if env var missing.                                         |
| POST /api/projects validation rejects E2E project data                 | Low        | Suite 03-06 project creation fails        | Use the exact same fields that already work in suite 03 (title, description, category, etc.)                                       |

| Test data accumulates in production DB                                 | Medium     | Noise in admin views and analytics        | `DELETE /api/admin/e2e/cleanup` endpoint called at end of each test run. Deletes all `@e2etest.invalid` users and related records. |

## Pre-Mortem

_Assume this plan failed. Most likely internal reasons:_

1. **Bootstrap problem with admin API key** (Task 3) → Trigger: `createE2EUser({ isAdmin: true })` needs an admin API key to set its own admin role, but the admin API key belongs to that user — circular dependency. **Mitigation:** The `E2E_ADMIN_API_KEY` env var points to a pre-existing admin user, not one created during the test run.

2. **Transaction seed doesn't satisfy review validation** (Task 6) → Trigger: `POST /api/reviews` returns 400 because the seeded transaction is missing a required field or the review service validates payment completion differently. **Mitigation:** Check `ReviewService` or `POST /api/reviews` route to see exactly which transaction fields are validated before implementation.

3. **Draft project approve flow doesn't match test expectations** (Task 6) → Trigger: `AdminService.approveProject()` on a draft project behaves differently than expected — maybe it doesn't set `status: 'active'` or requires additional fields. **Mitigation:** Read `ProjectRepository.approveProject()` during implementation to verify it sets both `status: 'active'` and `isApproved: true`.
